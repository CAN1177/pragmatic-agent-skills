# Launch Article Draft

## Pragmatic Agent Skills: 把 AI 工作方式沉淀成可复用的 skills

过去一段时间，我越来越不想把 agent 的能力沉淀成“提示词收藏夹”。

原因很简单。很多 prompt 在第一次使用时看起来有效，但一旦进入真实工作流，就会暴露几个问题：

- 触发条件不清楚
- fallback 行为不清楚
- 用户数据边界不清楚
- 下一次换个 agent 或换个机器，就很难稳定复用

所以我开始把这些经验换一种形式保存：不是存成 prompt，而是存成 skill。

我把这个仓库命名为 **Pragmatic Agent Skills**。它的目标不是做最大最全的 AI 工具箱，而是做一组足够窄、足够实用、足够可迁移的 skills。

每个 skill 都尽量回答这几个问题：

1. 它到底解决什么问题？
2. 什么情况下应该触发？
3. 什么情况下不该触发？
4. 如果输入不完整，fallback 顺序是什么？
5. 它和用户私有数据之间的边界在哪里？

这也是为什么这个仓库现在只保留了 `skills/` 作为核心结构。

## 这个仓库里现在有什么

目前先公开了 3 个 skill：

### 1. JobJD-Greeting-Generator

把 JD 和候选人真实信息压缩成一条可直接复制的招聘沟通话术。

它最重要的设计点不是“会写一句话”，而是：当用户给的是 `jd_url` 时，skill 不会一失败就立刻要求补全文，而是先按明确顺序做多路抓取和保守推断。

### 2. leetcode-coach

这个 skill 关注的不是题解本身，而是一个更底层的问题：公共 coaching 逻辑能否复用，而个人刷题记录能否始终保持私有。

所以它把仓库拆成三层：

- 公开 skill 仓库
- 本机中转工作区
- 用户私有远程记录仓库

### 3. self-improvement

很多 agent 都说自己会“学习”，但真正难的是：如何把一次次纠正，变成可追溯、可审核、可推广的改进候选项。

这个 skill 的设计就是先记事件，再蒸馏模式，再提出 promotion candidates，而不是一次反馈就直接回写规则。

## 我对 skill 的标准

我现在更关心这类 skill 是否满足几个工程上的要求：

- 足够窄，最好只解决一类问题
- 能说清楚边界，而不是泛化承诺
- fallback 可读，不藏在脑子里
- 能被别的 agent runtime 迁移
- 让另一个工程师看完就知道怎么复用

## 接下来会继续做什么

接下来这个仓库会继续补三类东西：

1. 新的高信号 skill
2. 更标准的 skill authoring template
3. 更轻量但明确的验证机制

如果你也在搭建 agent workflow，或者也在把自己的工作方式沉淀成可复用模块，这个仓库也许会对你有帮助。
